Initialisation§

$ mkdir git-introduction && cd git-introduction

$ git init                    # initialise a git repo!
Initialized empty Git repository in $HOME/git-introduction/.git/

$ ls -lah
drwxr-xr-x - slot  6 Aug 10:09 .git/

$ git config --global user.name "Tony Zorman"
$ git config --global user.email "tony.zorman@tu-dresden.de"

Without the -m option, git commit instead pops up an editor window in which to enter a commit message. Here, there’s also the option of specifying further comments after two additional line breaks: 
$ git commit
[main 28cb79a] Changed the first line
 1 file changed, 1 insertion(+), 1 deletion(-)
$ git log HEAD^..HEAD
commit 28cb79aca150f19b258732132f16ea0114c80a86 (HEAD -> main)
Author: Tony Zorman <soliditsallgood@mailbox.org>
Date:   Mon Aug 14 15:36:12 2023 +0200

    Changed the first line

    Another comment.

We still have unstaged changes. The index can be circumvented by supplying the -a flag to git commit. 
$ git commit -am "A bad commit message"
[main d4de7f6] A bad commit message
 1 file changed, 2 insertions(+), 3 deletions(-)

Branches§

Branches can also be created with git branch, but checkout immediately switches to them, so let’s use that instead. 
$ git checkout -b new-branch
Switched to a new branch 'new-branch'

To double check: 
$ git rev-parse --abbrev-ref HEAD
new-branch

Notice that git log correctly identifies HEAD pointing to new-branch now.
Changing something on the new branch§

Let’s change 2 Line to say something useless. 
$ emacs main.txt
$ cat main.txt
This is the first line!

1 Line
2 Line now says something useless
3 Line
4 Line
5 Line that says something else

This is the last line?

Committing the result, as we have done before. 
$ git commit -am "Change 2 Line to say something useless"
[new-branch f1aba65] Change 2 Line to say something useless
 1 file changed, 1 insertion(+), 1 deletion(-)
Merge conflicts§

How about we go back to the main branch, and make a similar change? 
$ git checkout main
Switched to branch 'main'

$ emacs main.txt
$ cat main.txt
This is the first line!

1 Line
2 Line says something useful!
3 Line
4 Line
5 Line that says something else

This is the last line?

$ git commit -am "Change 2 Line to say something useful"

If we try to merge new-branch into main now, we’ll get a merge conflict. 
$ git merge new-branch
Auto-merging main.txt
CONFLICT (content): Merge conflict in main.txt
Automatic merge failed; fix conflicts and then commit the result.

Fixing the conflict in an editor involves identifying the correct regions of text; these look like this: 
<<<<<<< HEAD
2 Line says something useful!
||||||| d4de7f6
2 Line
=======
2 Line now says something useless
>>>>>>> new-branch

We resolve the conflict by deciding upon a final form: 
2 Line says something even more useful!

and continue the merge. This will now prompt us for a commit message. 
$ git merge --continue
[main 3845de7] Merge branch 'new-branch'

We now get a pretty tree: 
$ git log --graph --pretty=oneline
\*   3845de7315908dbcd8b227b5f36fbddf779a950c (HEAD -> main) Merge branch 'new-branch'
|\
| * f1aba6569a83471ccad199985e0e5971505b0c16 (new-branch) Change 2 Line to say something useless
* | 5f21c5b97c0e243a06055b96a6b601ac4eec448e Change 2 Line to say something useful
|/
* d4de7f69083fcd893a0c176f3b4b0e355123dd66 A bad commit message
* 28cb79aca150f19b258732132f16ea0114c80a86 Changed the first line
* b38875afab25cefa61d7f4413cbbc0238d53b561 Initial commit

Remote repositories§

Creating a remote requires a Git forge; see the presentation. 
$ git remote add origin «URL»

Pushing code to this repository is done with git push; getting something off it works with git pull.

Merge conflicts can also happen while trying to pull from a remote repository. 
$ cd ~/ && git clone «URL» new-git-repo-name
$ cd new-git-repo-name
$ emacs main.txt # Change something.
$ git commit -am "A bad commit message"
$ cd ~/git-introduction # original repository
$ emacs main.txt # Change the same line in a different way.
$ git commit -am "Another bad commit message"
$ git pull       # Uh oh!

Often, you will also have unstaged changes when pulling; best to stash them. 
$ cd ~/ && git clone «URL» new-git-repo-name
$ cd new-git-repo-name
$ emacs main.txt # Change something.
$ git commit -am "A bad commit message"
$ cd ~/git-introduction # original repository
$ emacs main.txt # Change the same line in a different way
                 # NOTE: nothing added to index/no commit yet!
$ git stash
$ git pull
$ git stash apply

Other things to try out§

The following examples are taken from Git from the Bottom Up.

The many faces of git reset§

--mixed§
$ git add foo.c   # add changes to the index as a new blob
$ git reset HEAD  # delete any changes staged in the index
$ git add foo.c   # made a mistake, add it back
--soft§
$ git reset --soft HEAD^     # backup HEAD to its parent,
                             # effectively ignoring the last commit
$ git update-ref HEAD HEAD^  # does the same thing, albeit manually
--hard§
$ git reset --hard HEAD~3  # Go back in time, throwing away changes
$ git reset --soft HEAD~3  # Set HEAD to point to an earlier commit
$ git reset --hard  # Wipe out differences in the working tree

(almost) NEVER DO THIS! It’s much safer to 
$ git stash
$ git checkout -b new-branch HEAD~3   # head back in time!

Also 
$ git stash  # because it's always a good thing to do
$ git reset --hard HEAD~3  # go back in time
$ git reset --hard HEAD@{1}  # oops, that was a mistake, undo it!
$ git stash apply  # and bring back my working tree changes

In particular, HEAD@{1} accesses the reflog (git reflog). This is an incredibly useful tool anytime you think that you have lost some changes with Git—almost surely, you haven’t.
